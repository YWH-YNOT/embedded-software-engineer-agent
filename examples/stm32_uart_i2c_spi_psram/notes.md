# 示例说明

## 为什么这样设计

- 选 STM32，是因为它有完整的官方手册、HAL 与较常见的板级开发路径，适合展示“官方优先”的规则。
- 把 `SPI2 传输层` 和 `PSRAM 设备层` 分开，是为了体现“按模块拆解、分层验证”的原则。
- 刻意把 PSRAM 型号和温度传感器型号留到追问阶段，是为了展示 Agent 不能在关键资料缺失时直接写代码。
- 先做 UART 日志口，是为了给后续 I2C、SPI、PSRAM 调试提供最低成本的观测通道。

## 这个示例验证了什么

- Agent 会先审资料，而不是直接进入实现
- Agent 会主动追问少量但关键的问题
- Agent 会一次只推进一个模块
- Agent 会先给验证闭环，再宣称模块完成
- Agent 会在所有模块通过后才进入系统集成

## 这个示例没有试图覆盖什么

- 不覆盖 Linux BSP、驱动框架或复杂 GUI
- 不覆盖 RTOS 下的任务同步细节
- 不覆盖 DMA 优化、高吞吐调优、量产级压力测试
- 不试图给出具体芯片寄存器配置代码

## 如何复用到其他平台

你可以把这个流程迁移到其他平台，只要满足以下前提：

- 有明确芯片型号和板级连接信息
- 有官方参考手册 / 数据手册
- 有官方 SDK / HAL / 例程可参考
- 能为每个模块设计独立验证方法

平台可以换成 GD32、NXP、TI、ESP32、RP 系列等，但流程约束不变。
