# Skill: uart-bringup

## skill 名称

`uart-bringup`

## skill 描述

用于将 UART 模块从资料确认推进到最小可用 bring-up，并提供可执行的单模块验证路径。

## 触发场景

- 用户明确要做 UART / USART 初始化、日志口或基础收发
- 当前模块已收敛为单个 UART 实例
- 需要先验证 UART 是否可用，再决定是否做协议层、中断或 DMA 扩展

## 不应触发的场景

- 用户真正要解决的是上层串口协议，而不是底层 bring-up
- 芯片型号、引脚复用、时钟来源仍不明确
- 当前实际阻塞是调试工具、USB 转串口、地线连接或电平兼容问题但用户还未提供板级信息

## 所需输入

- MCU / SoC 型号
- UART / USART 实例号
- TX / RX 引脚与复用配置
- 系统时钟或外设时钟来源
- 波特率、数据位、校验位、停止位、流控要求
- 使用轮询 / 中断 / DMA 的约束
- 可用的串口工具或观测设备

## 推荐工作步骤

1. 先确认 UART 实例、引脚、时钟和目标用途。
2. 如果用途只是日志口，优先给出最小可用发送路径。
3. 如果存在协议收发需求，再判断是否需要 RX、中断、DMA 或缓冲区。
4. 先定义验证步骤，再讨论实现建议。
5. 要求模块通过单独验证后，才允许切换到下一个模块。

## 输出要求

- 当前只处理一个 UART 模块
- 说明依据来自哪些官方资料、HAL 或现有工程
- 至少给出 3 类验证项：基本发送、稳定性、波形或日志观测
- 明确不产出哪些内容，例如协议层、DMA 优化、环形缓冲

## 边界与风险

- 不能把串口工具收到字符当成全部验证结果
- 不能忽略波特率、时钟、地线和电平问题
- 不能在资料不足时编造 AF 映射或寄存器初始化顺序

## 验收方式

- 输出能直接指导 UART 最小 bring-up
- 验证清单里包含成功判据和失败排查路径
- 本轮边界明确，没有把协议层混进来
