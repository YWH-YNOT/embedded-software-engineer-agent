# Skill: project-intake

## skill 名称

`project-intake`

## skill 描述

用于把一个嵌入式项目从“模糊需求”收敛成“可进入资料审查”的输入包，优先明确平台、资料、约束、交付目标和阻塞项。

## 触发场景

- 用户第一次描述项目，希望 Agent 先审资料而不是直接写代码
- 用户提供了芯片型号、原理图、SDK/HAL、现有代码，但信息还很散
- 需要把自由文本整理进 `templates/project_intake.md` 和 `templates/hardware_intake.md`

## 不应触发的场景

- 当前已经进入单模块开发或单模块调试
- 用户的问题只针对某个已明确模块，例如仅问 UART 波特率异常
- 项目资料已经结构化完整，当前更需要模块计划或调试清单

## 所需输入

- 项目目标
- 芯片 / SoC 型号
- 原理图或板级连接信息
- 官方手册、数据手册、SDK/HAL、例程
- 现有代码、工具链、RTOS 情况
- 本轮希望交付什么

## 推荐工作步骤

1. 先识别平台、板卡版本、供电、时钟、调试接口、工具链和交付目标。
2. 将用户自由文本映射到 `templates/project_intake.md`。
3. 将板级事实映射到 `templates/hardware_intake.md`。
4. 标出 P0 缺失项，例如芯片子型号、原理图、关键器件型号、官方手册缺失。
5. 只提出会阻塞下一阶段的关键问题，不展开模块实现。

## 输出要求

- 使用仓库规定的 8 段输出结构
- 明确区分“已确认信息”和“缺失信息”
- 指出是否允许进入模块拆解
- 给出需要用户补充的最小资料清单

## 边界与风险

- 不能把口头描述当成板级事实
- 不能跳过资料审查直接进入模块开发
- 不能在资料接入阶段给出大量驱动实现细节

## 验收方式

- `templates/project_intake.md` 可被实际填写
- `templates/hardware_intake.md` 可被实际填写
- 输出中已明确 P0 阻塞项
- 用户看完后知道下一步该补什么，而不是拿到一堆泛泛问题
