# Skill: i2c-bringup

## skill 名称

`i2c-bringup`

## skill 描述

用于把 I2C 总线和单个目标器件的 bring-up 约束成受控流程，先验证总线，再验证设备寄存器或数据读写。

## 触发场景

- 用户要做 I2C 控制器初始化和单个传感器 / EEPROM / PMIC 的基本访问
- 用户已经提供了目标器件型号或地址信息
- 需要把总线 bring-up 与设备层验证分开

## 不应触发的场景

- 用户尚未明确 I2C 实例、引脚、器件型号或地址
- 当前问题是多设备总线仲裁、复杂驱动框架或 RTOS 异步架构
- 用户只想做板级资料接入，还没到模块开发阶段

## 所需输入

- MCU / SoC 型号
- I2C 实例号
- SCL / SDA 引脚和上拉信息
- 目标器件型号、7-bit 地址或数据手册
- 目标速率、是否需要 repeated start / clock stretching
- 验证工具和观测方式

## 推荐工作步骤

1. 先确认总线实例、引脚、电平、上拉和器件地址。
2. 先验证总线应答，再验证设备寄存器访问。
3. 明确区分“地址错误”“总线错误”“寄存器或字节序错误”三类问题。
4. 如果资料不足，优先追问器件型号、地址、寄存器页宽和时序要求。
5. 模块通过前，不扩展到多设备抽象或 DMA 版本。

## 输出要求

- 当前只处理一个 I2C 模块或一个 I2C 设备 bring-up
- 验证清单至少包含：寻址应答、寄存器读取、异常处理
- 明确指出本轮不处理的范围，例如多设备框架、DMA、RTOS 队列化访问

## 边界与风险

- 不能把器件地址或寄存器表猜出来
- 不能忽略上拉、电平和总线恢复
- 不能把一次成功读数当成模块稳定通过

## 验收方式

- 输出能指导完成总线层和设备层的最小闭环
- 验证项可在真实硬件上执行
- 失败时有清晰排查顺序
