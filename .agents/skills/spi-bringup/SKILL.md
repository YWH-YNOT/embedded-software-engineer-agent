# Skill: spi-bringup

## skill 名称

`spi-bringup`

## skill 描述

用于 SPI 控制器及其挂载设备的 bring-up，强调把“SPI 传输层”和“具体器件协议层”拆开处理。

## 触发场景

- 用户要做 SPI 总线 bring-up
- 用户要访问 SPI Flash、PSRAM、显示控制器或其他 SPI 从设备
- 需要先证明 SPI 时序和片选控制正确，再证明设备协议正确

## 不应触发的场景

- 未确认 SPI 实例、引脚、CPOL/CPHA、字长或片选关系
- 当前问题主要是高层缓存策略、文件系统或图形框架
- 用户尚未提供目标器件的官方数据手册

## 所需输入

- MCU / SoC 型号
- SPI 实例号
- SCK / MOSI / MISO / CS 引脚
- CPOL / CPHA、字长、位序、时钟上限
- 目标器件型号和官方数据手册
- 当前是否使用轮询 / 中断 / DMA
- 逻辑分析仪、示波器或日志通道

## 推荐工作步骤

1. 先确认 SPI 实例、模式、引脚和片选控制方式。
2. 先验证 SPI 事务层波形，再验证具体器件命令序列。
3. 对存储类器件，优先做固定地址读写与连续地址读写，不直接上性能调优。
4. 如果数据不一致，先分离为 SPI 模式错误、命令序列错误、片选 / 时序错误。
5. 模块验证通过前，不扩展到 DMA、缓存一致性或大吞吐优化。

## 输出要求

- 明确区分传输层与设备层
- 验证清单至少包含：波形检查、基础读写、连续地址访问
- 明确写出本轮不处理范围

## 边界与风险

- 不能跳过器件数据手册直接写命令序列
- 不能把波形正确等同于设备协议正确
- 不能把高层缓存管理混进 bring-up

## 验收方式

- 输出能指导 SPI 单模块 bring-up
- 失败排查路径能把总线问题和设备问题分开
- 用户能据此给出最小可复现验证记录
