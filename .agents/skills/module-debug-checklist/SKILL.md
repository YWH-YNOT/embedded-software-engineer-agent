# Skill: module-debug-checklist

## skill 名称

`module-debug-checklist`

## skill 描述

用于把单模块调试过程整理成结构化检查清单，适合在 bring-up 后、验收前或故障定位时触发。

## 触发场景

- 某个模块已经进入调试阶段，但现象、根因、修复记录还不够结构化
- 需要把调试和验收结果沉淀进 `templates/module_debug_checklist.md`
- 需要把“是否允许进入下一个模块或系统集成”写清楚

## 不应触发的场景

- 还处在资料审查阶段
- 当前尚未确定活动模块
- 用户需要的是新模块开发计划，而不是调试闭环

## 所需输入

- 当前模块名称
- 代码基线、板卡版本、测试人员和测试环境
- 已观察到的现象和失败条件
- 已做过的检查项和观测结果
- 目标成功判据

## 推荐工作步骤

1. 把当前模块映射到 `templates/module_debug_checklist.md`。
2. 先补齐测试前确认和硬件 / 软件前置检查。
3. 把现象转成测试用例，不要只写“有问题”。
4. 把排查顺序写成从硬件到底层配置再到协议层的路径。
5. 最终明确模块是否通过，以及是否允许进入下一个阶段。

## 输出要求

- 输出必须包含测试前提、测试项、实际结果、问题记录和验收结论
- 不得只给抽象建议而不落到检查表
- 明确是否允许进入下一个模块或系统集成

## 边界与风险

- 不能把“还没测”写成“通过”
- 不能只记录软件改动，不记录硬件和观测条件
- 不能跳过失败排查路径直接下结论

## 验收方式

- `templates/module_debug_checklist.md` 能被实际填写
- 调试记录能被其他开发者复现和审查
- 输出结论与当前阶段一致，不跨阶段偷跑
