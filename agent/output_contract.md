# Agent Output Contract

本文件定义 Agent 每次响应的固定结构。目标是让输出稳定、可审查、可复用、可追踪。

## 硬约束

- 每次输出都必须包含下列 8 个一级标题
- 标题名称与顺序保持不变
- 如果某一项当前为空，必须写“无”或“待补充”
- 不得在一个响应里同时推进多个未验证模块
- 结论必须能映射到当前阶段，不能跨阶段偷跑

## 每一节的含义

### 1. 当前阶段

必须明确当前处于哪个阶段，例如：

- `Phase 0: 资料审查`
- `Phase 1: 关键追问`
- `Phase 3: 单模块开发`
- `Phase 4: 单模块验证`

不能写成模糊表述，例如“先看一下”“继续推进”。

### 2. 已确认信息

只写已经确认的事实。每一项都应尽量标出依据来源，例如：

- 官方手册
- 官方 SDK/HAL
- 官方例程
- 用户提供的原理图
- 用户现有代码

如果只是经验推断，不应放在本节。

### 3. 缺失信息

只写当前仍缺失、且会影响判断或实现的内容。必须说明：

- 优先级
- 为什么重要
- 需要用户补充什么

不要把低价值问题堆满这一节。

### 4. 当前只处理的模块

必须明确：

- 当前唯一活动模块是什么
- 模块边界到哪里为止
- 依赖了哪些前置条件
- 哪些模块本轮不处理

这节的作用是防止范围失控。

### 5. 开发或调试建议

根据当前阶段填写：

- 如果还在资料审查或关键追问阶段，这里写审查结论与关键问题
- 如果已进入开发阶段，这里写实现建议、配置建议、风险提醒
- 如果在调试阶段，这里写故障定位顺序和排查建议

建议数量控制在 3 到 7 条，避免堆砌。

### 6. 验证清单

必须始终存在，即使当前还未开始执行，也应先给出计划中的验证项。每个验证项应包含：

- 怎么测
- 成功算什么
- 失败先查哪里

禁止使用“测试通过即可”这类空话。

### 7. 本轮产出与边界

要明确本轮真正产出了什么，以及明确不产出什么。例如：

- 本轮产出：模块拆解方案、UART 初始化建议、I2C 测试步骤
- 明确不产出：PSRAM 驱动代码、系统集成方案

### 8. 是否允许进入下一阶段

必须给出明确结论，只能是以下三种之一：

- `允许`
- `不允许`
- `有条件允许`

同时必须写清楚原因和前置条件。

## 典型错误

- 把“待确认信息”混进“已确认信息”
- 在“当前只处理的模块”里写多个活动模块
- 没有验证清单就宣布完成
- 没写清楚本轮边界，导致下一轮范围失控
- 用模糊词替代阶段结论
