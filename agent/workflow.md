# Agent Workflow

本文件把 Agent 的工作流写成受控阶段。可把它理解为一台状态机：只有当前阶段达到退出条件，才允许进入下一阶段。

## 状态流

`S0 资料审查 -> S1 关键追问 -> S2 模块拆解 -> S3 单模块开发 -> S4 单模块验证 -> 重复 S3/S4 直到模块完成 -> S5 系统集成 -> S6 回归测试与收尾`

## S0 资料审查

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 建立对项目边界、硬件条件、软件基础和现有资料的真实认识 |
| 输入 | 项目需求、芯片型号、原理图、引脚分配、官方手册、SDK/HAL、现有代码、板卡说明 |
| 输出 | 已确认信息清单、缺失信息清单、阻塞项列表 |
| 必要问题 | 芯片具体型号是什么；板卡版本是什么；时钟/供电是否已知；哪些资料已提供；是否存在现有工程基线 |
| 成功判据 | 能明确说出“现在知道什么、还不知道什么、哪些未知会阻塞开发” |
| 常见风险 | 芯片子型号搞错；引脚复用表不完整；板级约束未识别；把用户口述当成手册事实 |
| 是否允许进入下一阶段 | 仅当已确认信息与缺失信息被清晰分离时允许 |

## S1 关键追问

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 用尽量少的问题补齐真正影响开发路径的关键技术信息 |
| 输入 | S0 输出的缺失项与阻塞项 |
| 输出 | 按优先级排序的问题清单、用户补充后的约束 |
| 必要问题 | 影响架构、驱动写法、时序、中断、DMA、验证方式的问题 |
| 成功判据 | 关键阻塞项已解决，或已明确哪些模块必须暂停 |
| 常见风险 | 问题过多导致噪声；漏问 DMA/中断/验证条件；把次要问题放在关键问题前面 |
| 是否允许进入下一阶段 | 仅当当前模块所需的关键阻塞项已补齐时允许 |

## S2 模块拆解

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 把项目拆成可独立验证的模块，并排出依赖顺序 |
| 输入 | 用户目标、硬件资料、已补齐的关键约束 |
| 输出 | 模块列表、模块边界、依赖关系、开发顺序、当前活动模块 |
| 必要问题 | 哪些模块共享总线/中断/DMA；哪些模块依赖基础时钟或底层传输层；哪些模块可独立验证 |
| 成功判据 | 当前只剩一个活动模块，且其他模块被显式推迟 |
| 常见风险 | 模块边界过大；多个模块并行推进；未先做底层传输层就做上层驱动 |
| 是否允许进入下一阶段 | 仅当当前活动模块唯一且依赖条件明确时允许 |

## S3 单模块开发

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 基于官方资料和现有工程，完成当前模块的最小可用实现方案 |
| 输入 | 当前模块边界、官方手册、官方 SDK/HAL、官方例程、现有代码 |
| 输出 | 模块实现建议、接口定义、关键配置项、风险点 |
| 必要问题 | 该模块实例号是什么；引脚和时钟如何配置；使用轮询/中断/DMA 哪一种；和现有工程如何最小扰动集成 |
| 成功判据 | 实现方案清楚、依据清楚、验证方案已经预先定义 |
| 常见风险 | 在无依据情况下写寄存器；抽象过度；忽略现有工程约束；忽略初始化顺序 |
| 是否允许进入下一阶段 | 仅当验证方法和完成标准已写清楚时允许 |

## S4 单模块验证

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 证明当前模块在目标硬件与目标条件下可用 |
| 输入 | 当前模块实现方案、测试前提、仪器条件、验证脚本或测试步骤 |
| 输出 | 测试结果、成功判据匹配情况、失败排查路径、是否通过 |
| 必要问题 | 如何观测结果；如何区分硬件故障、配置错误、协议错误、时序错误；失败后先查哪里 |
| 成功判据 | 测试步骤可执行、结果可观测、结论可复现 |
| 常见风险 | 把“编译通过”当成功；只看单次现象不看稳定性；不区分驱动问题和板级问题 |
| 是否允许进入下一阶段 | 通过则允许切到下一个模块或进入集成；不通过则必须回到 S3 或 S1 |

## S5 系统集成

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 把所有已验证模块按受控方式集成到同一工程 |
| 输入 | 各模块通过记录、接口定义、资源占用、初始化顺序、任务/中断模型 |
| 输出 | 集成顺序、集成结果、冲突清单、系统级验证计划 |
| 必要问题 | 时钟树是否冲突；中断优先级是否冲突；DMA 是否冲突；缓存/内存分配是否冲突；初始化顺序是否正确 |
| 成功判据 | 关键功能可跑通，且已记录系统级耦合点和遗留风险 |
| 常见风险 | 单模块可用但集成后冲突；资源重用未记录；隐藏依赖未暴露 |
| 是否允许进入下一阶段 | 仅当系统级关键路径可运行时允许 |

## S6 回归测试与收尾

| 项目 | 内容 |
| --- | --- |
| 阶段目标 | 给出可执行的回归测试、风险清单和后续优化方向 |
| 输入 | 集成结果、遗留问题、性能数据、稳定性观察 |
| 输出 | 回归测试清单、风险清单、优化建议、版本边界说明 |
| 必要问题 | 哪些路径必须重复验证；哪些风险尚未关闭；哪些优化属于下一版本 |
| 成功判据 | 用户能据此继续测试、维护、交接或扩展项目 |
| 常见风险 | 只给结论不给清单；遗漏已知风险；把下一版本工作混入当前版本完成项 |
| 是否允许进入下一阶段 | 最终阶段，无后续状态；输出完整即可结束 |

## 回退规则

- 如果发现新的关键缺失信息，允许从 S3 / S4 回退到 S1。
- 如果发现模块边界划分错误，允许从 S3 回退到 S2。
- 如果集成中发现单模块验证不充分，必须回退到对应模块的 S4。

## 实践建议

- 每次响应只聚焦一个活动模块。
- 每个阶段都保留“输入、输出、成功判据”三件事。
- 任何“继续下一阶段”的决定都必须写清楚理由。
